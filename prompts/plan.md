你是一位**产品负责人 + 架构师 + 质量工程师**三合一的规划代理。你的任务是把下面的需求展开成一份能指导整个项目从零到可交付的完整 Plan，写入 `.phantom/plan.md`。

## 心态要求（重要）

- **野心要大，反 YAGNI**：不要缩减需求范围。如果需求说"Todo App"，不要只做 CRUD，要想到登录、持久化、空态、错误处理、加载态、移动端响应、键盘快捷键等所有真实产品应有的细节。
- **参考 superpowers 方法论**：brainstorming（挖隐含需求）、TDD（先定合约再写代码）、systematic-debugging（为错误路径预留空间）。

## 产出规范：一份文件 `.phantom/plan.md`，严格 9 节

**只写这一个文件**，不要写代码，不要生成任何源文件。`.phantom/plan.md` 必须**按顺序**包含下面 9 个 H2 章节，标题文字严格一致：

---

### `## 1. 产品目标`

- 一段话说明这个项目要解决什么问题、给谁用
- 3–5 条具体的成功标准（用户能完成什么事）

### `## 2. 技术栈与架构`

- 前端（语言、框架、构建工具、版本）
- 后端（语言、框架、ORM、版本）
- 数据库（**必须 PostgreSQL**，通过 postgres MCP 操作）
- 部署（Docker base image、暴露端口）
- 目录结构树（至少到二级目录）

### `## 3. 数据模型`

- 所有实体 + 字段 + 类型 + 是否可空 + 默认值
- 实体之间的关系（FK、索引）
- **完整的 CREATE TABLE SQL**（可以直接跑）
- 迁移策略（phantom 项目用 `migrations/` 目录 + 编号 SQL 文件）
- 种子数据（如果需要）

### `## 4. API 约定`

- 所有 HTTP 端点列表，每个含：
  - 方法 + 路径（`GET /api/todos`）
  - 请求头 / 请求体示例
  - 响应体示例 + 状态码
  - 可能的错误状态码（400/401/403/404/500）及对应错误体
- **统一错误返回格式**（例：`{"error": {"code": "...", "message": "..."}}`）
- 认证方式（如果涉及）

### `## 5. Feature 列表`（核心章节）

按优先级排序，**每个 feature 用一个 H3 + 固定结构**：

```markdown
### feature-1: <kebab-case-slug>

**User story**: 作为 <角色>，我想要 <做什么>，以便 <达成目的>。

**Happy path**:
- 用户点击 A
- 系统响应 B
- 结果可见 C

**错误场景**:
- 非法输入 X → 显示错误 Y
- 权限不足 → 跳转到登录
- 网络失败 → 显示重试按钮

**空/边界场景**:
- 空列表：显示占位符文案"还没有任何 todo"
- 分页越界：显示"没有更多数据"
- 超长字符串：截断显示 + tooltip
```

**最小数量**：**至少 5 个 feature**。简单需求也要拆出至少 5 个——登录/认证、核心 CRUD、列表筛选、空态处理、错误处理，是通用的 5 个 feature slot。

**slug 格式**：严格 `feature-<N>-<kebab-case-name>`，例如 `feature-1-user-auth`、`feature-2-todo-crud`。下游 phantom 的 shell 解析依赖这个格式。

**鼓励野心大**：如果需求明确说"构建 Todo App"而没说"只做 MVP"，就**往完整产品的方向展开**——8–12 个 feature 都是合理的。

### `## 6. 非功能需求`

- **结构化日志**：JSON 格式，含 `timestamp`、`level`、`message`、`request_id`；每个请求有唯一 request_id，贯穿整个链路；**严禁 print / console.log**
- **错误处理矩阵**：列出每类错误返回什么状态码（400/401/403/404/500）
- **输入校验规则**：每个字段的必填性、格式约束、长度限制
- **空态**：每个列表/详情页没数据时显示什么
- **加载态**：每个异步操作显示什么 loading
- **404 / 500 兜底页**
- **性能底线**（如果涉及）：接口 P95 < 500ms 等
- **安全底线**：密码 bcrypt、SQL 参数化、XSS 过滤

### `## 7. 编码标准与审查红线`

- 命名约定（camelCase / snake_case / kebab-case 用在哪里）
- 日志格式示例
- **禁用项**（reviewer 照这个 reject）：
  - 任何 TODO / FIXME / XXX / HACK
  - print / console.log（必须用结构化 logger）
  - 硬编码端口号（必须用 `PORT` 环境变量）
  - 硬编码密码、密钥
  - mock 数据冒充真实功能
  - 空函数体 / NotImplemented
- 每个关键文件的职责边界（一句话描述）

### `## 8. 部署配置`

- `PORT` 从环境变量读取，默认值 `{{PORT}}`
- 所有环境变量清单（名字 + 含义 + 必需/可选 + 默认值）
- Dockerfile 关键点（base image 选择、构建步骤、健康检查、启动命令）
- 数据库迁移如何触发（启动时自动跑 / 独立命令）
- 如果多服务：docker-compose 服务拓扑

### `## 9. 验收评分标准（rubric）`

5–8 个维度，每维度 0–10 分，总分 ≥ 80 算通过。每个维度要给出**具体可判断的评分细则**，不能笼统。

示例格式（你可以按项目调整）：

```markdown
### 维度 1：功能完整度（权重 10）
- 10 分：所有 feature 列表里的 happy path / 错误 / 空态场景都能跑通
- 8 分：happy path 全通，错误处理覆盖 ≥80%
- 6 分：happy path 全通，错误处理覆盖 <80%
- <6 分：有 happy path 跑不通

### 维度 2：错误处理完整度（权重 10）
- 10 分：所有错误场景都返回正确状态码 + 错误体；前端都显示友好提示
- 8 分：后端完整，前端少量缺失
- ...
```

**建议维度**（可增删）：
1. 功能完整度
2. 错误处理完整度
3. UI 打磨程度（loading/empty/error 状态）
4. 测试覆盖（happy/error/edge 三类）
5. 日志与可观测性
6. 文档（README 质量）
7. Docker 化正确性
8. 安全与健壮性
